Creation, Props & Self

Cette documentation nécessite une connaissance relative de ReactJS.

ReasonReact n'utilise pas/n'a pas besoin de classes. L'API de création de composants vous retourne un simple record, dont vous pouvez surcharger les champs (par exemple render).

Le modèle de composant est créé par le biais de ReasonReact.statelessComponent "TheComponentName". La chaîne de caractères est utilisée à des fins de debugging uniquement (l'équivalent du displayName de ReactJS).

À titre d’exemple, voici le fichier Greeting.re :

let component = ReasonReact.statelessComponent "Greeting";

In ReactJS, you'd create a component class and call it through JSX which transforms into React.createElement(myClass, {prop1: 'hello'}) under the hood. En ReasonReact, au lieu de passer la "classe" complète (ie modèle de composant) dans une hypothétique fonction ReasonReact.createElement, nous allons plutôt définir une fonction make :

/* toujours dans Greeting.re */
let component = ReasonReact.statelessComponent "Greeting";

let make ::name children => {
  ...component, /* propage les autres valeurs défaut du modèle ici  */
  render: fun self => <div> (ReasonReact.stringToElement name) </div>
};

La fonction make est la fonction appelée par le JSX de ReasonReact, décrite plus tard. Pour le moment, la manière non-JSX d'appeler et de faire le rendu d'un composant est :

ReasonReact.element (Greeting.make name::"John" [||]) /* la fonction `make` du module `Greeting` */
/* l'équivalent de <Greeting name="John" /> */

Il est nécessaire que make retourne le record de composant défini ci dessus. Vous pouvez surcharger quelques champs, tels que les familiers render, initialState, didMount, etc., tous décrits plus loin.

Note: ne pas inliner let component dans le corps de la fonction make comme ce qui suit !

let make _children => {...(ReasonReact.statelessComponent "Greeting"), render: fun blabla}

Puisque make est appelé à chaque invocation JSX, vous créériez accidentellement un nouveau composant à chaque fois.

Props #

Les propriétés sont juste des arguments nommés de la fonction make, vue plus haut. Ils peuvent également être optionnels et/ou avoir des valeurs par défaut, par exemple let make ::name ::age=? ::className="box" _children => ....

La dernière propriété doit être children. Si vous ne l’utilisez pas, ignorez la simplement en la nommant _ ou _children. Les noms commençant par 'underscore' ne déclencehent pas d'alertes lorsqu'ils sont inutilisés.

Le nom d'une propriété ne peut pas être ref ni key. Ce sont des noms réservés, tout comme en ReactJS.

Following that example, you'd call that component in another file through <Foo name="Reason" />. className, if omitted, defaults to "box". age defaults to None. If you'd like to explicitly pass age, simply do so: <Foo name="Reason" age=20 />.

Neat Trick with Props Forwarding #

Parfois en ReactJS, vous recevez une propriété du parent que vous souhaiteriez transmettre directement à l’enfant :

<Foo name="Reason" age={this.props.age} />

C’est une source d'erreurs, parce que this.props.age pourrait être accidentellement changé en un nombre nullable alors que Foo n’attend pas qu’il le soit, ou vice versa ; il pourrait être nullable avant, et maintenant qu'il ne l'est plus Foo se retrouve avec un contrôle de nullité inutile quelque part dans la fonction de rendu.

En Reason, si vous voulez passer de façon explicite une propriété ageFromProps optionelle (dont le type est option int, c'est à dire None | Some int), ce qui suit ne fonctionnerait pas:

<Foo name="Reason" age=ageFromProps />

Because age expects a normal int when you do call Foo with it, not an option int! Naively, you'd be forced to solve this like so:

switch ageFromProps {
| None => <Foo name="Reason" />
| Some nonNullableAge => <Foo name="Reason" age=nonNullableAge />
}

Pénible. Heureusement, il y a une meilleure façon de passer explicitement une valeur optionnelle :

<Foo name="Reason" age=?ageFromProps />

It says "I understand that age is optional and that when I use the label I should pass an int. But I'd like to forward an option value explicitly". This isn't a JSX trick we've made up; it's just part of the language feature! See the section on "Explicitly Passed Optional" in the Reason docs.

self #

You might have seen the render: fun self => ... part in make. The concept of JavaScript this doesn't exist in ReasonReact (but can exist in Reason, since it has an optional object system); the this equivalent is called self. It's a record that contains state, retainedProps, handle and reduce, which we pass around to the lifecycle events, make and a few others, when they need the bag of information. These concepts will be explained later on.
